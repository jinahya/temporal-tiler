== Architecture

=== Current Design (after refactor)

* **`TemporalTiler`** — `final` utility class. Single static method: `tile(T, T, U, UnaryOperator<T>, TileFactory<T, U>)`. No class-level type parameters, no state, no inheritance.
* **`TemporalTile<T, U>`** — abstract base for a single tile (`[start, end)` + grain + aligned flag). `T extends Temporal & Comparable<? super T>`, `U extends TemporalUnit`.
* **`ChronoTile<T>`** — concrete `TemporalTile<T, ChronoUnit>`.
* **`ChronoTiler`** — `final` utility class. Single static method: `tile(T, T, ChronoUnit)`. Delegates to `TemporalTiler.tile()` with built-in `ChronoUnit` truncation and `ChronoTile` factory.

=== What was removed

* `TemporalTiler.Builder` / `ChronoTiler.Builder` — no longer needed; ranges are method arguments, not stored state.
* Class-level `T` on tilers — `T` is now method-level, inferred from arguments.
* `TemporalTiler` as abstract base class with `truncate()` / `newTile()` abstract methods — replaced by `UnaryOperator<T>` and `TileFactory<T, U>` parameters.
* `& Serializable` bound on `T` — unnecessary; tiling logic never serializes temporal values.

== What Works Well

=== Focused scope
Does one thing (range decomposition) and nothing else. No feature creep.

=== Correct algorithm
The truncate + iterate approach in `TemporalTiler.tile()` is sound. Half-open intervals `[start, end)` eliminate boundary ambiguity.

=== Type safety
`T extends Temporal & Comparable<? super T>` is the right constraint. Supports all major `java.time` types: `LocalTime`, `LocalDate`, `LocalDateTime`, `OffsetDateTime`, `ZonedDateTime`.

=== Solid test coverage
Tests across five temporal types with edge cases: empty ranges, smaller-than-grain, gap-free verification, hierarchical chaining.

=== Clean API surface
Two entry points: `ChronoTiler.tile(start, end, grain)` for standard use, `TemporalTiler.tile(start, end, grain, truncator, factory)` for custom `TemporalUnit` types. No builders, no instance creation.

=== Parameterized Type Usages

==== Inheritance

* `ChronoTile<T>` extends `TemporalTile<T, ChronoUnit>`
* `TemporalTileAssert<T, U>` extends `AbstractAssert<TemporalTileAssert<T, U>, TemporalTile<T, U>>`

==== Unchecked casts to `(T)` in `ChronoTiler.truncate()`

All casts from `Temporal` API return values back to `T`:

* `(T) value.with(ChronoField.NANO_OF_DAY, truncated)` -- time-based truncation
* `(T) result.with(ChronoField.NANO_OF_DAY, 0L)` -- zero out time fields
* `(T) result.minus(dayOfWeek - 1, ChronoUnit.DAYS)` -- WEEKS truncation
* `(T) result.with(ChronoField.DAY_OF_MONTH, 1)` -- MONTHS
* `(T) result.with(ChronoField.DAY_OF_YEAR, 1)` -- YEARS
* `(T) result.with(ChronoField.YEAR, ...)` -- DECADES, CENTURIES, MILLENNIA

These casts are safe in practice because `Temporal.with()`/`minus()` return the same concrete type, but the compiler cannot verify this.

==== Concrete type arguments in tests (`Chrono_Tile_Test`)

* `ChronoTiler.tile(LocalTime, ...)` -- time-based tests (HOURS, MINUTES)
* `ChronoTiler.tile(LocalDate, ...)` -- date-based tests (DAYS, WEEKS, MONTHS, YEARS)
* `ChronoTiler.tile(LocalDateTime, ...)` -- combined date-time tests
* `ChronoTiler.tile(OffsetDateTime, ...)` -- offset-aware tests
* `ChronoTiler.tile(ZonedDateTime, ...)` -- zone-aware tests

=== Observations

* `T` lives at class level only on `TemporalTile` (which holds typed `start`/`end` fields) -- correct, because the tile owns data of type `T`.
* `T` lives at method level on both tilers -- correct, because the tiler is stateless and infers `T` from arguments. A single `ChronoTiler.tile()` call works with any compatible temporal type.
* `U` on `TemporalTiler.tile()` is truly generic: callers can supply any `TemporalUnit` with a custom truncator. `ChronoTiler` fixes `U = ChronoUnit` and provides the truncator.
* No `& Serializable` on `T` -- the tiling logic never serializes temporal values. `TemporalTile` implements `Serializable`, and all standard `java.time` types are serializable, so serialization works in practice without the bound.

== Bottom Line

The core algorithm (~50 lines in `tile()` + `truncate()`) is correct and useful. The architecture now matches the complexity of the problem: two static methods, one abstract tile class, one concrete tile class. Extension via `TemporalTiler.tile(T, T, U, truncator, factory)` is available for custom `TemporalUnit` types without requiring subclassing.
