== Architecture

=== Current Design

* **`TemporalTile<T>`** — `final` immutable tile representing a half-open range `[start, end)` at a `TemporalUnit` grain. Single type parameter `T extends Temporal & Comparable<? super T>`. No inheritance, no serialization concern.
* **`TemporalTiler`** — `final` utility class. Two static `tile()` overloads:
** `tile(T, T, TemporalUnit, UnaryOperator<T>)` — general-purpose, caller supplies truncator.
** `tile(T, T, ChronoUnit)` — convenience, built-in truncation for all standard `ChronoUnit` values.

=== What was removed

* `ChronoTile<T>` — unnecessary subclass; `TemporalTile` is now `final` and concrete.
* `ChronoTiler` — absorbed into `TemporalTiler` as a convenience `tile(T, T, ChronoUnit)` overload.
* `TemporalTiler.TileFactory` — no longer needed; `TemporalTiler` creates `TemporalTile` directly.
* `U extends TemporalUnit` type parameter — grain is stored as `TemporalUnit`; no practical benefit from parameterizing the unit type.
* `& Serializable` bound on `T` — tiling logic never serializes temporal values.
* `implements Serializable` on `TemporalTile` — not the library's concern.
* `TemporalTiler.Builder` / `ChronoTiler.Builder` — ranges are method arguments, not stored state.
* Class-level `T` on tilers — `T` is method-level, inferred from arguments.
* `TemporalTiler` as abstract base class with `truncate()` / `newTile()` abstract methods — replaced by `UnaryOperator<T>` parameter.

== What Works Well

=== Focused scope
Does one thing (range decomposition) and nothing else. No feature creep.

=== Correct algorithm
The truncate + iterate approach in `TemporalTiler.tile()` is sound. Half-open intervals `[start, end)` eliminate boundary ambiguity.

=== Type safety
`T extends Temporal & Comparable<? super T>` is the right constraint. Supports all major `java.time` types: `LocalTime`, `LocalDate`, `LocalDateTime`, `OffsetDateTime`, `ZonedDateTime`.

=== Solid test coverage
Tests across five temporal types with edge cases: empty ranges, smaller-than-grain, gap-free verification, hierarchical chaining.

=== Minimal API surface
Two classes, one type parameter. `TemporalTiler.tile(start, end, ChronoUnit.MONTHS)` for standard use, `TemporalTiler.tile(start, end, grain, truncator)` for custom `TemporalUnit` types. No builders, no factories, no instance creation.

=== Parameterized Type Usages

==== Inheritance

* `TemporalTileAssert<T>` extends `AbstractAssert<TemporalTileAssert<T>, TemporalTile<T>>`

==== Unchecked casts to `(T)` in `TemporalTiler.truncate()`

All casts from `Temporal` API return values back to `T`:

* `(T) value.with(ChronoField.NANO_OF_DAY, truncated)` -- time-based truncation
* `(T) result.with(ChronoField.NANO_OF_DAY, 0L)` -- zero out time fields
* `(T) result.minus(dayOfWeek - 1, ChronoUnit.DAYS)` -- WEEKS truncation
* `(T) result.with(ChronoField.DAY_OF_MONTH, 1)` -- MONTHS
* `(T) result.with(ChronoField.DAY_OF_YEAR, 1)` -- YEARS
* `(T) result.with(ChronoField.YEAR, ...)` -- DECADES, CENTURIES, MILLENNIA

These casts are safe in practice because `Temporal.with()`/`minus()` return the same concrete type, but the compiler cannot verify this.

==== Concrete type arguments in tests

* `TemporalTiler.tile(LocalTime, ...)` -- time-based tests (HOURS, MINUTES)
* `TemporalTiler.tile(LocalDate, ...)` -- date-based tests (DAYS, WEEKS, MONTHS, YEARS)
* `TemporalTiler.tile(LocalDateTime, ...)` -- combined date-time tests
* `TemporalTiler.tile(OffsetDateTime, ...)` -- offset-aware tests
* `TemporalTiler.tile(ZonedDateTime, ...)` -- zone-aware tests

=== Observations

* `T` lives at class level only on `TemporalTile` (which holds typed `start`/`end` fields) -- correct, because the tile owns data of type `T`.
* `T` lives at method level on the tiler -- correct, because the tiler is stateless and infers `T` from arguments. A single `TemporalTiler.tile()` call works with any compatible temporal type.
* No `U extends TemporalUnit` parameter -- grain is stored as `TemporalUnit` in the tile. The unit type parameter added no practical value (no standard `TemporalUnit` implementations beyond `ChronoUnit`; callers never needed the concrete unit type in the return).
* No `& Serializable` on `T` -- the tiling logic never serializes temporal values. All standard `java.time` types are serializable, so serialization works in practice without the bound.

== Bottom Line

The core algorithm (~50 lines in `tile()` + `truncate()`) is correct and useful. The architecture is minimal: two classes, one type parameter, zero inheritance. Extension via `TemporalTiler.tile(T, T, TemporalUnit, truncator)` is available for custom `TemporalUnit` types without requiring subclassing or factory interfaces.
